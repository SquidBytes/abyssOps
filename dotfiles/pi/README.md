# Pi setup — abyssOps dotfiles

Project-agnostic Pi scaffolding for the GSD workflow. Clone the abyssOps repo on any machine; this tree provides the *universal* part (skills, rules, base system prompt). Project-specific data lives in a manifest at each project's `.pi/project.yaml`, generated by the `gsd-install-project` skill.

## What's in here

| Path | Purpose |
|---|---|
| `INSTALL.md` | Two-stage install: global toolkit, then per-project bootstrap. Start here. |
| `system-prompt/` | Profile-level prompt overlays Pi loads at session start. Pi-agnostic and GSD-aware halves. |
| `skills/` | GSD verbs (discuss / plan / execute / debug / commit-guard / install-project) plus the shared `note-marker` skill (`/marknotes`). All read project-specific data from `.pi/project.yaml`. |
| `rules/` | Universal rule fragments referenced by skills. |
| `templates/` | Manifest schema + skeletons for adding new skills, rules, or overlays. |
| `examples/` | Worked examples of real project manifests and overlays — read these to understand the shape, not copy verbatim. |

## How project-specific behavior flows

```
~/.pi/agent/
├── AGENTS.md                       # base.md + gsd-overlay.md, composed at install
├── skills/                         # symlinked or copied from dotfiles/pi/skills/
├── rules/                          # symlinked or copied from dotfiles/pi/rules/
└── extensions/                     # Pi plugins (powerline, todo, rewind, …)

<project-root>/
└── .pi/
    ├── project.yaml                # generated per-project by gsd-install-project
    ├── AGENTS.md                   # project-specific overlay; layered on top of global
    └── rules/
        └── pii-policy.md           # only if pii.enabled in the manifest
```

Skills read `<project-root>/.pi/project.yaml` and resolve paths / rules from it. Two projects on the same machine can have completely different shapes (mono vs nested, GSD vs not, different commit conventions) and the same skills work in both.

## Design choices worth knowing

- **Plugins do the primitives, skills do the workflow.** Statusline, context warnings, checkpoint/rewind, structured user-questions, and persistent todos all come from Pi packages (see `INSTALL.md`). Skills in this tree only encode workflow logic specific to GSD.
- **Soft enforcement first.** Skills ask Pi to "stop and ask" or "verify before commit"; they don't hard-block tool calls. If softness proves insufficient for a given rule, the fallback is a Pi hook or extension.
- **Layered system prompt.** `base.md` is Pi-agnostic, `gsd-overlay.md` activates when a project has `.pi/project.yaml` with a non-null `planning.root`. Per-project AGENTS.md adds the project's own prose context.
- **Manifest before skills.** Skills check for `.pi/project.yaml` on first invocation. If missing, they direct the user to run `gsd-install-project` rather than guess at paths.
- **Frontmatter records destination.** Every file in this tree has a `pi_install:` block at the top telling you where it goes. `INSTALL.md` aggregates them into one checklist.

## Iterating

When something feels off in Pi's behavior:

1. Identify which file (skill / rule / overlay) controls it.
2. Edit it here in `dotfiles/pi/`.
3. Reinstall by re-running the relevant step in `INSTALL.md` (or `cp` / symlink directly).
4. Reload Pi (`/reload` if running).

When you want to add a new workflow verb, copy `templates/skill-template/`. When you want a new rule, copy `templates/rule-template.md`. When you set up a new project, run the `gsd-install-project` skill — it does the manifest + overlay + PII-rule scaffolding for you.
